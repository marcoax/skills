This article isn't meant to be a ranking or a competition between tools. There's no winner here. The goal is much simpler: to share a real-world case where the solution came from an unexpected direction, and to remind ourselves that when facing a complex bug, locking ourselves into a single approach is often the most effective way to stay stuck.

The Problem

After migrating from DocumentDB 3.6 to DocumentDB 5, on a PHP 7.2 / Symfony 3.4 / alcaeus/mongo-php-adapter 1.1.8 stack, this warning started appearing on every MongoDB operation (insert, update, deleteMany):

bson_append_array(): invalid array detected. first element of array parameter is not "0"

Key characteristics of the bug:

  • Present only on AWS DocumentDB 5, not on local MongoDB
  • Queries executed correctly — no data lost
  • array_values() on the array passed to the query didn't fix anything
  • Logs confirmed the array had sequential keys starting from 0

The First Instinct: StackOverflow

As almost always when encountering an obscure bug, the first move was to search StackOverflow. In this thread about MongoDB/DocumentDB bson_append_array() error, the suggested solution was to add &authMechanism=SCRAM-SHA-1 to the connection string. A change that, honestly, seemed unlikely to work — what does the authentication mechanism have to do with a BSON serialization error? It looked like one of those "it works but nobody knows why" solutions, and that wasn't enough. So I decided to truly understand the problem and turned to AI.

More or less all AI agents (Claude, Codex, Kimi, Qwen) proposed the solution summarized below.

Debugging with AI

The debugging journey followed this sequence:

1. Non-sequential array hypothesis — verified and discarded through granular logging directly in the code. The array had sequential keys starting from 0, array_values() changed nothing.

2. $in workaround with single element — replacing $in with direct match for single-element arrays. Didn't fix it.

3. Explicit normalization — array_filter + array_map('strval') + array_values(). Didn't fix it.

4. Warning suppression with restore_error_handler:

set_error_handler(function() {}, E_WARNING);
$this->collection->deleteMany(['_id' => ['$in' => $ids]]);
restore_error_handler();

It worked as a workaround but masked the problem without solving it.

AI Conclusion: None of the proposed solutions eliminated the warning. The correct diagnosis was that the bug resided in the alcaeus layer with DocumentDB 5, and the definitive fix required a deeper migration — removing alcaeus and migrating to the native mongodb/mongodb driver.

Back to the Beginning

After half a day of debugging — and more — with no concrete results, with the warning still there and none of the AI solutions actually eliminating it, that moment every developer knows well had arrived: the existential phase. The one where you start questioning not just the code, but your entire career. Twenty years of experience and a PHP warning sends you into crisis. Maybe I should have become an accountant.

And it was precisely in that moment of healthy desperation that I decided to play the "all or nothing" card: go back to the starting point and try that StackOverflow solution I had smugly dismissed at the beginning because it seemed too far-fetched to be true. After all, what did I have to lose? I'd already left my dignity somewhere between the third var_dump and the fifth array_values().

The StackOverflow Solution

A StackOverflow thread suggested adding to the connection string:

&authMechanism=SCRAM-SHA-1

One line. Warning gone. Faster queries as a side effect.

Why It Works

SCRAM-SHA-256 is the default authentication mechanism on DocumentDB 5. It requires more round-trips during the handshake compared to SCRAM-SHA-1. This changes how the driver negotiates capabilities with the server, and alcaeus in legacy mode behaves differently in BSON serialization — eliminating the warning as a side effect.

It's not a logically direct solution, it's empirical. But it works.

The Lesson

Systematic debugging with AI led to:

  • Correctly identifying where the problem resided (alcaeus layer)
  • Confirming that queries worked despite the warning
  • Defining the definitive long-term fix (removing alcaeus)

The StackOverflow solution resolved the symptom in an unexpected and non-obvious way, with a performance bonus.

Neither approach is "better" — they're complementary. Systematic debugging provides understanding of the problem, the community provides battle-tested empirical solutions. When facing a bug, keeping an open mind to both sources is often the difference between solving a problem and truly understanding it.
